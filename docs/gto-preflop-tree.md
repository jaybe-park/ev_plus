## 왜 프리플랍은 "전부"가 가능한가

포스트플랍이 전부 수집 불가인 이유는 **보드 카드 조합 폭발**(플랍만 1,755 텍스처 /
정확 보드 128만+)이다. 프리플랍은 커뮤니티 카드가 없고 **순수 액션 시퀀스**만
상황을 정의하므로 트리가 유한하고 열거 가능하다. 따라서 텍스처 클래스 근사가
필요 없이 "진짜 전부"를 목표로 잡아도 된다.

169핸드 매트릭스는 키에 넣지 않는다 — GTO Wizard가 노드당 전체 169핸드
전략을 한 번에 주므로, 기존 프리플랍 방식과 동일하게 데이터(핸드 빈도) 쪽에 포함된다.

## GTO Wizard 트리 구조 실측 (③ 정찰 결과, 2026-07-13)

Browser 페인으로 `Cash6mGeneral_6mNL25R25`, depth=100 스팟들을 직접 열어 확인.

### 핵심 발견 1 — 노드당 논-올인 레이즈 사이즈는 정확히 1개
각 의사결정 노드에서 가능한 액션은 **{Fold, Call(합법일 때), Raise <단일 사이즈>, Allin}**.
"레이즈 사이즈 여러 개"가 아니라 노드마다 **딱 하나의 레이즈 사이즈 + 올인**만 제공됨.
- RFI(UTG): Raise 2.5 / Allin / Fold
- vs_open(HJ vs UTG 오픈): Raise 8 / Call / Allin / Fold
- 콜드-4벳(CO, UTG오픈+HJ3벳 뒤): Raise 17.5 / Allin / Fold (콜 없음)
- 5벳(BTN, …4벳 뒤): Raise 35 / Allin / Fold

→ 트리의 분기 폭발이 "베팅 사이즈"에서 오지 않는다. 분기는 **누가 fold/call/raise/allin
하느냐**에서만 온다. 이것이 트리를 유한·소규모로 유지하는 핵심.

### 핵심 발견 2 — 리레이즈 깊이 (100bb 기준)
사이즈 에스컬레이션: **2.5(오픈) → 8(3벳) → 17.5(4벳) → 35(5벳) → 올인(6벳)**.
100bb에서 명시 레이즈 약 4단계 후 올인이 강제된다(5벳 35 이후 다음 공격은 사실상 올인).
스택 깊이가 바뀌면 이 단계 수도 바뀌지만, 우리는 **항상 100bb 가정**을 유지한다
(헤즈업 포함, docs/gto-data.md의 근사 원칙과 동일).

### 핵심 발견 3 — 스퀴즈/콜드콜/콜드-4벳 노드가 전부 존재하고 solvable
"UTG 오픈 → HJ 3벳 → CO 콜드-4벳" 같은 노드가 GTO Wizard에 실제로 있음(위에서 확인).
현재 우리 모델(오프너 본인만)이 스킵하던 바로 그 상황들이 여기서 채워진다.

### 핵심 발견 4 — 콜 옵션의 등장/소멸
Call은 vs_open(오픈에 대한 플랫콜)에는 있으나, 4벳+ 노드에는 없어짐(Fold/Raise/Allin만).
파서/스키마는 노드마다 가능한 액션 집합이 다름을 전제해야 한다.

### 핵심 발견 5 — 사이즈는 노드/포지션 의존, 공식 추론 불가
같은 "오픈에 대한 3벳"이라도 포지션마다 사이즈가 다름(기존 확인: HJ 3벳=8, SB=11, BB=13.5).
**모든 사이즈는 수집 시 화면에서 실측**해야 하며 계산/보간 금지.

### URL = 자연스러운 캐노니컬 키
GTO Wizard URL이 전체 경로를 그대로 인코딩한다:
`preflop_actions=R2.5-R8-R17.5-F-F&history_spot=N`
(F=fold, C=call, R{bb}=raise to, 포지션 순서 UTG→HJ→CO→BTN→SB→BB로 매핑,
history_spot=N=지금까지 나온 액션 수=현재 액션할 좌석의 결정 시점 인덱스).
→ 이 `preflop_actions` 문자열이 그대로 우리 노드 키가 된다.

## 스키마 설계 (②)

### 키 = 액션 시퀀스 문자열
현재 `(position, vs_position, range_type)` enum 3종 키를
**정규화된 액션 시퀀스 문자열**(GTO Wizard `preflop_actions` 포맷)로 교체.
- 예: 현재 "HJ vs UTG open" = `R2.5-F` (UTG 2.5 오픈, HJ 결정 시점) → history_spot=1
- 예: 현재 "UTG vs HJ 3bet" = `R2.5-R8` → history_spot=2 (오프너 UTG가 3벳에 대응)
- 지금까지 모은 11~35개 조합은 포지션 순서/range_type 의미에서 시퀀스 문자열을
  **결정론적으로 역산 가능** → 마이그레이션 가능, 버릴 데이터 없음.

### 테이블 방향 (구현 시 확정)
- 안 A: `gto_preflop_situations`에 `action_seq TEXT` 컬럼 추가 + UNIQUE(action_seq).
  기존 (position, vs_position, range_type)은 조회 편의/디버깅용으로 파생 유지 or 폐기.
- 안 B: 신규 `gto_preflop_nodes(action_seq PK, hero_position, num_active, raise_size)`
  + 핸드 빈도는 기존 `gto_preflop_hands`를 새 FK로 재사용.
- 어느 쪽이든: `hero_position`, `num_active`(참여 인원)는 시퀀스에서 파생해 인덱스/쿼리용
  컬럼으로 둔다. `raise_size`는 그 노드에서 히어로가 레이즈할 때의 실측 to-금액(bb).

### 사이징 그래뉼래리티 (조회 시 필수 처리)
실전(사람/봇)의 베팅은 연속값인데 솔브 트리는 노드당 이산 사이즈 1개.
실시간 조회 시 **실제 시퀀스를 캐노니컬 노드로 스냅**해야 한다:
각 레이즈를 "그 노드의 캐노니컬 사이즈"로 반올림 매핑(예: 실전 3벳 7.5bb → 노드 8 트리로).
스냅 규칙은 ② 구현 시 정의(가장 가까운 캐노니컬 사이즈, 없으면 미수집 큐로).

## ② 구현 결정 및 완료 (2026-07-15, 순수 코드)

### 택한 스키마 = 안 A (저위험, additive)
`gto_preflop_situations`에 `action_seq TEXT`(캐노니컬 노드 키) + `hero_position TEXT`
+ `num_active INTEGER`(파생) 컬럼을 **추가**. 기존 `(position, vs_position, range_type)`
enum 컬럼과 UNIQUE는 **그대로 유지**(레거시/파생, nullable). 노드 키 유니크는 별도
부분 유니크 인덱스 `idx_gto_pre_seq`로 강제(SQLite ADD COLUMN이 UNIQUE 인라인 불가 +
nullable 다수 NULL 허용). `SCHEMA_VERSION` 11→12.
- **안 B(신규 gto_preflop_nodes 테이블) 반려 이유**: 핸드 빈도 FK 재배선/조회부 이중화
  비용이 크고, 기존 35개 enum 경로·봇·테스트를 건드릴 표면이 넓어짐. 안 A는 컬럼 추가만으로
  두 조회 경로를 병렬 공존시켜 "additive & non-breaking" 원칙에 가장 부합.

### 캐노니컬 스냅 = **레이즈 깊이 기준**(사이즈-근접 아님) — 핵심 결정
> ⚠️ **2026-07-16 대체됨**: 아래 "깊이-캐노니컬 사이즈를 저장 노드 키에 쓴다"는
> 결정은 **데이터 기반 수집 방식**(이 문서 하단 "데이터 기반 트리 수집" 섹션)으로
> 대체됐다. 저장 키는 이제 **실측 사이즈**를 쓰고, 깊이-캐노니컬 스냅은 **런타임
> 조회에서 트리에 없는 사이즈를 매칭할 때만** 폴백으로 남는다. 아래는 ② 시점의
> 기록으로 보존(변경 이력).

`gto/url_generator.py`에 단일 소스로 정의:
```
CANONICAL_OPEN_SIZE      = {UTG:2.5, HJ:2.5, CO:2.5, BTN:2.5, SB:3.5}   # 깊이1(오픈), 포지션 의존
CANONICAL_RAISE_BY_DEPTH = {1:2.5, 2:8.0, 3:17.5, 4:35.0}               # 3벳=8, 4벳=17.5, 5벳=35
```
`canonical_raise_size(depth, position)` — 저장(백필/save)과 런타임(advisor)이 **같은 함수**를
호출. 스냅은 베팅 **금액**이 아니라 **레이즈 순번(깊이)**으로 한다.
- **왜 사이즈-근접이 아니라 깊이인가**: 실측 3벳이 포지션마다 8/11/13.5bb로 흩어지는데,
  단순 "가장 가까운 캐노니컬 사이즈"로 스냅하면 13.5bb 3벳이 4벳(17.5)에 더 가까워
  **깊이 오분류**가 발생한다. 깊이는 액션 순서로 구조적으로 결정되므로 이 함정을 원천 차단.
- **트레이드오프**: 3벳+ 깊이는 포지션 의존 실제 사이즈(11/13.5)를 단일 캐노니컬(8)로 **뭉갠다**.
  이는 의도된 근사 — 저장 노드 키와 런타임 조회 키가 동일 테이블을 쓰므로 항상 매칭되는 것이
  최우선(키 생성 일원화). 노드의 히어로 **자기** 레이즈 사이즈(4벳 실측 등)는 `raise_size`
  컬럼에 그대로 보존되므로 봇 사이징 정보는 손실 없음(노드 키에만 스냅 적용).

### 노드 키 규약
- 노드 키 = 히어로가 결정하기 **직전까지**의 액션 시퀀스(자기 이번 액션 제외).
  GTO Wizard `preflop_actions` + `history_spot`(=토큰 수) 규약과 동일.
- 토큰: `F`=fold, `X`=check, `C`=call, `R{bb}`=raise/allin(깊이 캐노니컬 사이즈).
  포지션 순서 UTG→HJ→CO→BTN→SB→BB. RFI UTG = `""`(빈 문자열).
- 생성 단일 소스: `url_generator.situation_to_node_key(enum)`(백필/save 파생) ↔
  `advisor.canonical_node_key(preflop_seq)`(런타임 스냅). 두 함수는 같은 캐노니컬 사이즈
  테이블을 공유 → 결정론적으로 동일 문자열 생성.

### ④(수집)로 넘길 때 주의점
- 수집 노드 키는 `situation_to_node_key` 또는 실전 `canonical_node_key`가 만든 **캐노니컬
  스냅 키**로 저장해야 런타임 조회와 매칭됨(GTO Wizard 화면 실측 사이즈를 그대로 키에 넣지 말 것).

- `save` 엔드포인트는 `action_seq`를 직접 받거나(임의 노드) enum에서 파생 가능. 스퀴즈/멀티웨이
  같은 enum-불가 노드는 `action_seq`를 명시해 저장하면 시퀀스 경로로만 조회된다.

## 실시간 조회를 위한 선행 조건 (①) — 코드 조사 결과

**게임 진행 중 advisor가 쓸 구조화된 액션 히스토리가 현재 코드에 없다.**
- `core/game.py`의 `event_log`: `_emit("action", {player, action(한글), amount})`로
  기록되나 **position/street 필드가 없고, 아무도 소비 안 함**(dead code에 가까움).
  포지션은 `get_positions()`(이름→라벨, 헤즈업은 "BTN/SB")로 별도 조회해야 함.
- 실제 동작 경로는 `server/session.py`의 `action_log: List[str]`(한글 텍스트,
  예 "UTG 레이즈 2.5bb")이고, `gto/advisor.py`가 이를 **문자열 부분매칭**으로

## 트리 노드 수 실측 계산 (2026-07-16, 열거 + 브라우저 검증)

전체 트리를 열거해 "수집 대상 의사결정 노드 수"를 계산. **콜드콜 규칙이 규모를
지배**한다(브라우저 실측으로 확정):
- 오픈(레벨1)에 직면: fold/call/raise/allin (콜드콜=오픈 콜 허용 → 멀티웨이 가능)
- **3벳+(레벨≥2)에 직면: 콜드 플레이어는 콜 불가**(fold/raise/allin만). 이미 자발
  투자한 사람(오프너)만 디펜스 콜 가능. (실측: UTG오픈-HJ3벳에서 CO(콜드)=콜 없음,
  오프너 UTG=콜 있음 → "invested만 콜" 규칙)
- 자발 레이즈 깊이 상한 4(open→3bet→4bet→5bet), 그 이상 공격은 allin.

이 규칙으로 열거한 노드 수 (멀티웨이 상한 M = 동시 자발참여 인원):

| 목표 범위 | 노드 수 | 무료한도(100/일) | 자동화(≈6초/노드) |
|---|---|---|---|
| 헤즈업만 (M=2) | 220 | ~2.2일 | ~22분 |
| ≤3-way (M=3) | 2,665 | ~27일 | ~4.4시간 |
| ≤4-way (M=4) | 16,831 | ~168일 | ~28시간 |
| 전체 멀티웨이 (M=6) | 91,794 | ~2.5년 | ~6.4일 |

⚠️ 초기 오산: 콜드콜-3벳(존재하지 않는 가지)까지 포함해 519,372로 과다계산했던 것을
브라우저 실측으로 바로잡음(→ 91,794). 열거 스크립트: scratchpad/count_v2.py, count_v3.py.

**노드당 실측 비용**(3개 샘플): 수동(모델 구동, 스크린샷 생략) ≈ 33초/노드 + 3~6k
토큰/노드. 자동화(Playwright, 로컬 스크립트) ≈ 6초/노드 + LLM 토큰 0.

**결론**: "전부(9.2만)"는 가성비 최악(2.5년/6.4일). **헤즈업 220노드가 1차 목표**
(실전 대부분 커버, 수동 ~2h 또는 자동화 22분) → ⑤ 검증 → ≤3-way(2,665)로 확장.
지배 병목은 무료 한도(100/일) + 수동 토큰 → M≥3은 ④ 자동화 필수, 유료 플랜이면 대기 소멸.

## 데이터 기반 트리 수집 방식 (2026-07-16 확정) — ② 대비 변경점 포함

트리를 "가정"으로 열거하지 않고, **GTO Wizard 데이터가 트리를 스스로 정의하게** 한다
(lazy, reachability-driven expansion). 규모 사전 계산은 불필요·불가(모든 노드를 방문해야
알 수 있음) — 그래서 규모 계산은 접는다.

### 원칙
1. **가정 금지**: "항상 4액션" / "콜드콜 없음" 같은 규칙을 코드에 박지 않는다.
   각 노드에서 GTO Wizard가 실제로 보여주는 액션·사이즈로만 가지를 뻗는다.
2. **빈도 기반 분기(ε=0.05%)**: 한 노드에서 **레인지 전체 콤보 가중 합산 빈도가
   0.05%를 초과하는 액션**만 자식 가지로 확장. (버튼 존재 ≠ 분기 — 균형에서 아무 핸드도
   안 하는 액션은 실전 도달 0에 가까워 스킵. ε은 CSS 추출 반올림 먼지 + 미미한 가지 배제용.)
3. **저장 키 = 실측 사이즈 그대로**: 노드 키의 각 레이즈는 화면에서 읽은 실제 to-금액(bb)로
   표기(예 BB 3벳 노드 = `R2.5-F-F-F-F-R13.5`, `R8`로 뭉개지 않음). GTO Wizard는 노드당
   사이즈 1개라 결정론적 → 실측 키가 안정적. **스냅은 저장 시점엔 불필요.**
4. **스냅은 런타임 조회 때만**: 실전에서 상대가 트리에 없는 사이즈로 베팅하면(예 7.5bb 3벳),
   그 시점에 **수집된 트리의 그 위치에서 가장 가까운 실측 사이즈 노드로 스냅**. 스냅에 쓰는
   "가능한 사이즈 목록"은 하드코딩 테이블이 아니라 **수집된 데이터에서 읽는다**.
5. **수집 우선순위 — GTO 트리 체계 수집이 항상 1순위**:
   - **1순위 = GTO 데이터 기반 트리 전체를 체계적으로 수집.** 단순 깊이 BFS가 아니라
     **액션 빈도(도달 확률) 가중 best-first**로 확장한다: 각 노드의 도달 확률(경로상 액션
     빈도들의 누적, 콤보 가중)이 높은 가지부터 먼저 수집 → 부분 수집 상태여도 실전에서
     자주 나오는 라인이 먼저 채워진다. (구현 시 도달확률 지표 정의; 얕고 빈발한 노드 우선.)
   - **2순위 = 큐(GTO 트리에서 벗어난 스팟).** 아레나/플레이 중 트리에 없는 스팟이
     나오면 큐에 쌓고, 나중에 수집. 큐 수집 시엔 그 노드만 찍지 말고 **경로상 조상 노드 +
     빈도>ε 자손 노드(프리플랍 연속)** 를 함께 수집(이미 근처까지 navigate 했으므로 재방문
     =일일 한도 추가 소모를 아낌). 단 **큐는 어디까지나 보조** — 체계 수집(1순위)이
     빈도 가중이라 자주 나오는 스팟은 이미 대부분 커버되고, 큐는 그 밖의 엣지를 메운다.

### ② 대비 무엇이 바뀌나 (코드 영향 분석)

**핵심 전환**: ②는 노드 키를 **깊이-캐노니컬 사이즈**(3벳=항상 `R8`)로 저장하고 저장·런타임이
동일 하드코딩 테이블을 공유해 "항상 매칭"시켰다. 데이터 기반은 **실측 사이즈 키** + **런타임
트리-인지 스냅**으로 바꾼다. 영향 범위(파일별):

### 확정 결정 D1~D3 (2026-07-16, 사용자 승인 — 권장안대로 확정)

- **D1 — 하드코딩 캐노니컬 테이블: 노드 키 생성에서 제거.**
  `url_generator`의 `CANONICAL_OPEN_SIZE`/`CANONICAL_RAISE_BY_DEPTH`/`canonical_raise_size`/
  `_canon_raise_token`은 **저장·런타임 노드 키 생성 경로에서 뺀다**. 저장 키는 실측 사이즈,
  런타임 스냅은 **수집된 형제 노드의 사이즈**를 읽어 매칭한다. 맞는 형제가 없으면 하드코딩
  값으로 추측하지 않고 **큐로 보낸다**(추측 금지 대원칙과 일치). 테이블은 즉시 삭제하지 않고
  참조가 모두 끊긴 것을 확인한 뒤 제거(또는 "미사용/레거시" 주석만 남기고 데드코드 정리).

  위배. 스냅 근거를 데이터에 두는 것이 가정 제거라는 이 설계의 핵심과 일치.

- **D2 — 레거시 11노드: 데이터 기반 워커로 재수집(v13 마이그레이션 안 함).**
  ②가 백필한 캐노니컬-사이즈 키 11행은 ④ 워커가 트리를 걸을 때 **실측-사이즈 키로 자연히
  덮어써진다**(얕은 코어라 수집 비용 낮음 + 워커 정확성 회귀 대조까지 겸함). 그때까진 캐노니컬
  키가 임시로 남되, D1/스냅 로직이 "수집 형제 기준"이라 잘못된 조회를 유발하지 않는다.
  별도 v13 재키잉 마이그레이션은 만들지 않는다(중복 작업 회피).
  *근거*: 어차피 전 트리를 재수집하므로 별도 마이그레이션은 낭비. 재수집이 워커 검증도 됨.

- **D3 — 런타임 스냅 규칙: "그 프리픽스에 수집된 레이즈-형제로 매핑, 없으면 큐+휴리스틱".**
  런타임에 라이브 시퀀스로 키를 만들 때, 각 라이브 레이즈를 **그 지점(프리픽스)에서 수집된
  자식 중 레이즈-형제의 사이즈로 스냅**한다. GTO Wizard 트리는 노드당 레이즈 사이즈가 1개라
  보통 형제가 하나뿐 → 그걸로 매핑(수치 거리 컷 불필요). 만약 한 노드에 레이즈 사이즈가
  2개 이상 수집돼 있으면 **bb 절대거리 최소**로 선택. 해당 프리픽스/브랜치가 아직 수집 안 됐으면
  숫자로 억지 매칭하지 말고 **큐에 등록 + equity 휴리스틱 폴백**.
  *근거*: 노드당 사이즈 1개라는 ③ 실측 덕에 "허용 오차 임계값"이 대부분 불필요해지고,
  미수집 브랜치는 추측 대신 큐로 보내는 게 데이터 기반 원칙에 부합.

### 서브에이전트 지시 요지 (구현 위임 시 반드시 전달)
1. ①(structured seq)·스키마 컬럼·큐·save의 action_seq 수신은 **재사용, 건드리지 말 것**.
2. 저장 노드 키는 **실측 사이즈**로 생성(깊이-캐노니컬 금지). GTO Wizard 화면에서 읽은
   실제 사이즈를 키 토큰에 사용.
3. 분기 기준은 **버튼 존재가 아니라 레인지 합산 빈도 > ε(0.05%)**.
4. 런타임 조회 스냅은 **수집된 트리의 형제 노드 사이즈**를 기준으로(하드코딩 테이블 아님),
   매칭 실패 시 **큐로**(추측 채움 금지).
5. 모든 변경은 기존 enum 경로·봇·테스트가 통과하는 선에서(가능하면 병렬 공존), 실패 시 중단.
   `tests/run_all.py --full` 통과 게이트. 통과 시에만 커밋(푸시는 사용자).
6. 규모를 미리 세지 말 것 — 큐 빌 때까지/일일 한도까지 데이터 기반으로 걷는다.

## ④ 드라이버 구현 완료 (2026-07-17) — `scripts/collect_gto_tree.py`

데이터 기반 트리 워커의 **Playwright(CDP) 자동화 드라이버**를 구현했다. 순수 로직
(`scripts/gto_tree_worker.py`: `aggregate_frequencies`/`branch_actions`/`action_to_token`/
`TreeNode`/`FrontierQueue`)은 **그대로 재사용**하고, 이 파일은 거기에 브라우저 조작과
저장을 붙인 얇은 드라이버다.

### 드라이버가 하는 일 (노드 1개 처리)
1. `url_from_node_key(node_key)`로 GTO Wizard로 navigate(실측 사이즈 키 → URL 정확 일치).
2. 레인지 셀(`[data-tst^="range_table_cell_0_"]`) 렌더 대기: **색칠된 셀 개수가 600ms 이상
   안 바뀌면 완료로 판정**(고정 임계값 아님 — 아래 "라이브 실측 확정" 2026-07-17 버그 수정 참고).
3. 검증된 CSS 파서(`colorToAction`/`parseCell`)로 169핸드 액션 빈도 추출.
4. **badSum 검증**: 핸드별 fold+call+raise+allin 합이 [0.9,1.1] 밖이면 저장 안 하고 스킵
   (failed에 기록, 진행 계속).
5. 화면 텍스트에서 raise/allin **실측 사이즈** 스크레이프(`/Raise ([\d.]+)/` 등).
6. `POST /gto/preflop/save`에 **action_seq=실측 사이즈 키 verbatim**(D1), raise_size 실측,
   hands 저장(기존 엔드포인트 재사용, 변경 없음).
7. 저장된 hands로 `aggregate_frequencies`+`branch_actions`(ε=0.05%) → 자식 노드를
   **도달확률 가중**(부모 reach × 액션 빈도)으로 `FrontierQueue`에 push. 베팅 종료
   (터미널) 자식은 `_replay` 시뮬레이터로 걸러 제외(무효 navigate 방지).

### 핵심 설계 준수
- **저장 키는 실측 사이즈**(깊이-캐노니컬 하드코딩 금지, D1). 추측 채움 없음.
- **분기는 빈도>ε**(버튼 존재로 판단 금지). `branch_actions` 단일 소스.
- **우선순위는 도달확률 best-first**(자주 나오는 라인 먼저). `FrontierQueue`.
- **베팅순서 시뮬레이터**(`_replay`)는 순수 포커 규칙(누가 다음에 행동/라운드 재개/
  터미널)만 쓴다 — GTO "가정"이 아님. 이걸로 히어로/레이저 포지션과
  `hero_position`/`vs_position`/`range_type`/`situation_label`을 결정론적으로 유도.
  DB의 기존 13노드 라벨과 100% 일치 확인.

### 사용자 실행 방법

드라이버는 **사용자가 미리 로그인해 둔 GTO Wizard 크롬 세션에 CDP로 붙는다** — 로그인
자체를 대신 하지 않는다(자격증명 안전). **아래 사전 조건이 매번 갖춰져 있어야 실행된다.**

#### ✅ 실행 전 체크리스트 (매번 확인)
1. **디버그 크롬 창이 떠 있고, 그 창에서 GTO Wizard(`app.gtowizard.com`)에 로그인돼 있어야 함.**
   기존에 평소 쓰던 크롬이 열려 있으면 먼저 전부 종료한 뒤(같은 프로필로 두 개 띄우면
   디버그 포트가 활성화 안 되는 경우가 실제로 있었음 — 그래서 반드시 `--user-data-dir`로
   별도 프로필 사용), 터미널에서:
   ```
   /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
       --remote-debugging-port=9222 \
       --user-data-dir="$HOME/chrome-gto-debug"
   ```
   뜬 창에서 `https://app.gtowizard.com` 접속 → 로그인. **이 창을 실행 내내 켜둔 채로 유지.**
2. **백엔드 서버가 떠 있어야 함**(수집 결과를 저장할 곳): `./start.sh` 실행,
   `https://localhost:8765/docs`가 응답하는지 확인.
3. 확인 커맨드(선택, 문제 진단용):
   ```bash
   curl -s http://localhost:9222/json/version   # 디버그 크롬 응답 확인
   curl -sk https://localhost:8765/docs -o /dev/null -w "%{http_code}\n"  # 백엔드 확인
   ```
   9222가 응답 없으면 1번을, 8765가 응답 없으면 2번을 다시 확인.

#### 실행
4. **파서 눈검증(dry-run, 최초 1회 권장)**: 저장 없이 첫 노드만 추출해서 보여줌
   ```bash
   python3 scripts/collect_gto_tree.py --dry-run
   ```
5. **실전 수집**: `--limit`을 크게 잡아 "무료 일일 한도(100/일)에 도달할 때까지" 쭉 돌려도
   안전하다(한도 도달 시 자동으로 안전 종료, 처리 중이던 노드도 유실 없이 다음 실행에
   이어짐 — 아래 "라이브 실측 확정" 버그 수정 참고).
   ```bash
   python3 scripts/collect_gto_tree.py --limit 200
   ```
   (한도가 보통 100/일이라 `--limit`을 그보다 크게 줘도 문제없음 — 안전마진에서 자동 정지됨.
   **하루 지나 한도가 초기화되면 그냥 같은 커맨드를 다시 실행하면 체크포인트에서 이어감.**)
6. 실행 중 중간에 Ctrl+C로 중단해도 안전 — 마지막 저장 지점까지 체크포인트 보존, 재실행하면 이어감.
7. 끝나면 확인: `python3 scripts/audit_gto_preflop.py` (전수검사), DB 스팟 수 늘었는지 확인.

### CLI 옵션
- `--limit N`(기본 90) 이번 실행 최대 신규 노드 수. 일일 한도(100/일)보다 크게 줘도
  안전마진에서 자동 정지되므로, "한도까지 쭉 돌리고 싶다"면 넉넉히(예: 200) 줘도 됨.
- `--dry-run` 추출/검증만, 저장·확장 없이 첫 노드 상세 출력.
- `--min-delay`/`--max-delay`(기본 2.0/5.0초) 노드 처리 사이 랜덤 지연 — 기계적으로
  너무 빠른 요청 패턴을 피하기 위한 배려(2026-07-17 추가). 필요시 조정 가능.
- `--cdp-url`(기본 `http://localhost:9222`) 크롬 CDP 엔드포인트.
- `--server`(기본 `https://localhost:8765`) 저장 대상 FastAPI(자체서명 → 검증 스킵).
- `--checkpoint`(기본 `<repo>/gto_tree_checkpoint.json`) 진행상황 JSON.
- `--epsilon`(기본 0.0005) 분기 빈도 컷. `--nav-timeout`(기본 30000ms) — 렌더 대기가
  안정화 기반이라 대부분 실제로는 몇 초 내로 끝남(깊은 노드도 마찬가지), 값 자체는
  "최악의 경우 최대 대기"일 뿐.
- `--safety-margin`(기본 5) 일일 한도까지 남은 여유가 이 값 이하면 새 수집을 시작하지 않고
  안전 종료.

### 중단-재개 / 체크포인트
- 노드를 저장할 때마다 `gto_tree_checkpoint.json`에 `visited`/`frontier`/`failed`를 원자적
  기록(.tmp→rename). 재실행 시 체크포인트가 있으면 프론티어를 그대로 복원, 없으면 DB의
  기수집 트리를 루트부터 BFS로 훑어 **미수집 자식만 도달확률 가중으로 재구성(seed)**.
- DB의 `action_seq` 존재 여부로도 최소 재개가 되지만(중복 수집 방지), 체크포인트가
  프론티어 재계산 비용을 없앤다. `.gitignore`에 등록(로컬 상태, 커밋 대상 아님).
- **일일 한도 도달 시에도 유실 없음**(2026-07-17 버그 수정): 처리 중이던 노드가 한도에
  걸리면 그 노드를 다시 큐에 되돌려 넣고 안전 종료 — 다음 실행(내일 등)에서 정확히
  그 지점부터 재개된다. (수정 전엔 이 노드가 visited에도 frontier에도 없이 사라지는
  버그가 있었음 — 사용자가 라이브 테스트로 지적해 발견/수정.)

### 라이브 실측 확정 (2026-07-17) — 초기 추측 로직 2건 교체

첫 라이브 테스트(사용자 CDP 세션)로 아래 두 부분이 **실측 DOM 신호로 확정**됐다.
(이전 "추측 기반" 구현이 만든 버그 2건을 함께 수정.)

- **실측 사이즈 = 히어로 "Actions" 패널 버튼 (data-tst 스코프)**. 이전 구현은
  `body.innerText` 전체에서 `Raise N`을 정규식으로 잡아, **헤더 카드(다른 포지션이
  이미 취한 과거 액션, 예 "CO 100 Raise 2.5")를 히어로 레이즈로 오인**했다(버그:
  BTN vs CO open 노드에서 raise_size가 8이어야 하는데 2.5로 추출). 확정된 소스:
  컨테이너 `[data-tst="study_action_btns"]` 안의 `[data-tst^="action_"]` 버튼들만
  히어로가 지금 취할 수 있는 액션이다. 버튼 `data-tst`가 액션+사이즈를 인코딩:
  - `action_RAI_<i>` = 올인(to-amount는 버튼 텍스트 "Allin N"에서 파싱, 못 읽으면 100bb 폴백)
  - `action_R<size>_<i>` = 레이즈 to `<size>` (예 `action_R8_1` = Raise 8, `action_R17.5_1` = Raise 17.5)
  - `action_C_<i>` = 콜, `action_F_<i>` = 폴드
  코드의 숫자는 화면 표시 텍스트("Raise 8" 등)와 **verbatim 일치**하고, 헤더
  (`hspotcrd_action_text`)·핸드테이블 셀(`htc_action_*`)과 접두어가 달라 **절대 겹치지 않는다**.
  라이브 대조 4노드 전부 일치 확인: UTG RFI=2.5, BTN vs CO open=8, HJ vs UTG open=8,
  CO vs HJ 3bet=17.5.

- **일일 한도 = "X/100" 카운터가 유일한 권위 신호**. 이전 구현은 추측 키워드 목록
  (`daily limit`/`무료`/`한도` 등)으로 감지했다. 실측 결과 상단에 `X/100` 사용량
  카운터가 항상 노출되고(스팟 새로 볼 때마다 +1: 18→19→20 실측), 그 옆
  "Free accounts can browse 100 preflop spots per day." 문구는 **한도와 무관하게 늘
  떠 있는 툴팁**이라 단독 판단 불가. → 카운터를 파싱해 남은 여유(100−X)를 계산하고,
  안전마진(`--safety-margin`, 기본 5) 이하로 떨어지면 **다음 navigate 전에 안전 종료**
  (체크포인트 보존 → 재실행 시 이어감). 카운터를 못 읽을 때만 문구+렌더실패를 폴백으로 사용.
  루프 진입 전에도 현재 열린 페이지에서 사용량을 스팟 소모 없이 미리 확인한다.

- Playwright 번들 크로미움은 CDP connect엔 필수 아님(사용자 크롬에 붙음). 단
  `python3 -m playwright install chromium`을 해두면 드라이버 바이너리가 준비돼 안정적
  (이 저장소에선 설치 완료).

### 두 번째 라이브 실행 확정 (2026-07-17) — 실전 버그 2건 추가 수정

첫 실전 실행(`--limit`)에서 발견/수정:

- **렌더 대기 절대 임계값 버그**: "색칠된 셀 ≥150개"를 렌더완료 기준으로 썼는데, 4벳+
  같은 깊은 노드는 오프너의 계속 레인지가 원래 좁아(예 88/169 — 나머지는 오프닝 레인지
  밖이라 정당한 `background:none`) 150을 영원히 못 넘겨 **무한 타임아웃**(60초→90초로
  늘려도 실패)했다. "로딩 중"과 "이미 다 됐는데 원래 좁은 레인지"를 구분 못 하는 게
  근본 원인. 수정: 절대 개수가 아니라 **색칠된 셀 개수가 600ms 이상 안 바뀌면 렌더 완료**로
  판정(`polling=200ms`, 매 노드마다 안정성 상태 리셋). 같은 실패 노드가 수정 후 4.4초 만에
  정상 추출(88핸드, raise_size=23) 확인.
- **일일 한도 도달 시 노드 유실 버그**(사용자 지적): `frontier.pop()`으로 꺼낸 노드가
  `limit_hit`이면 다시 큐에 안 들어가고 `break`만 해서, `visited`에도 `frontier`에도 없는
  상태로 체크포인트에 저장돼 사실상 영구 유실됐다(체크포인트가 있으면 다음 실행 때 DB
  reseed를 안 하므로 다시 나타날 기회가 없음). 수정: `limit_hit` 시 `frontier.push(node)`로
  되돌려 넣은 뒤 중단. 몽키패치로 한도초과 시뮬레이션해 재현·검증(프론티어 유실 없음 확인).
- **랜덤 지연 추가**(사용자 제안): 노드 처리 사이 `--min-delay`/`--max-delay`(기본 2~5초)
  균등분포 랜덤 대기. 기계적으로 빠른 요청 패턴이 어뷰징으로 감지/차단될 위험을 줄임.

이 두 실행(`--limit 5` 3회)으로 DB가 13→24개로 늘었고, `audit_gto_preflop.py`/
`tests/run_all.py --fast` 전부 통과 확인.

## 수집·검증 전략 (④⑤⑥)

- **④ 수집 엔진 = 데이터 기반 트리 워커**(위 "데이터 기반 트리 수집" 참조): 노드의 제공
  액션·실측 사이즈·합산 빈도를 화면에서 추출 → ε 초과 액션만 자식으로 확장.
  **1순위=GTO 트리 체계 수집(빈도 가중 best-first, 도달확률 높은 라인부터), 2순위=큐**
  (트리 밖 스팟, 이웃 동시 수집). 규모 사전열거 없이 트리 소진/일일 한도까지 걷는다.
  Chrome 수동 수집은 토큰상 수십 개까지만 현실적 → Playwright 자동화가 사실상 필수.
- **⑤ 파일럿+검증**: 전체 크롤링 전에 스퀴즈/멀티웨이 소수 노드만 수집 →
  아레나 bb/100으로 "equity 휴리스틱 대비 개선" 확인(포스트플랍 계획과 동일 게이트).
  실전 볼륨 대부분(싱글레이즈/싱글3벳)은 이미 35개로 커버되므로 이득은 롱테일에 집중.
- **⑥ 본격 확장**: 파일럿에서 개선 확인 시 트리 전체로 확장. 개선 없으면 롤백.
